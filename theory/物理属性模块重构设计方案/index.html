    物理属性模块重构设计方案¶

    1. 当前设计存在的问题¶

    1.1 高耦合问题¶

    当前 SinglePhaseProperties 类直接操作 mesh.cell_list 中的单元属性，违反了封装原则。物理属性计算与网格数据结构紧密耦合，导致难以独立测试和维护。

    1.2 职责不清晰¶

    SinglePhaseProperties 类既负责物理属性的存储和计算，又负责更新网格单元的属性，职责过于集中。

    1.3 冗余数据存储¶

    对于简单的均匀属性情况，存储整个三维数组是不必要的内存开销。特别是在1D或2D问题中，多个方向的渗透率可能是冗余的。

    1.4 缺乏扩展性¶

    当前设计难以支持复杂的非均匀属性场，如从文件读取的渗透率场或通过地质统计学方法生成的属性场。

    2. 新设计方案¶

    2.1 设计目标¶

    1. 降低模块间的耦合度
    2. 明确各模块的职责
    3. 提高代码的可测试性和可维护性
    4. 增强系统的可扩展性

    2.2 核心设计思路¶

    2.2.1 引入属性管理器模式¶

    创建专门的属性管理器来处理物理属性的存储和访问：

    2.3.3 属性管理器¶

    class PropertyManager:
    """属性管理器"""

    def __init__(self, mesh: StructuredMesh, config: Dict[str, Any]):
        self.mesh = mesh
        self.properties = self._initialize_properties(config)

    def get_cell_property(self, cell_index: int, property_name: str) -> Union[float, np.ndarray]:
        """获取指定单元的属性值"""
        if property_name in self.properties:
            prop = self.properties[property_name]
            if isinstance(prop, np.ndarray):
                # 根据网格维度获取属性
                coords = self.mesh.get_cell_coords(cell_index)
                return prop[coords]
            else:
                return prop
        else:
            raise ValueError(f"Property {property_name} not found")

    def _initialize_properties(self, config: Dict[str, Any]) -> Dict[str, Any]:
        """初始化所有物理属性"""
        properties = {}
        properties['porosity'] = self._init_porosity(config)
        properties['permeability'] = self._init_permeability(config)
        return properties

    3. 优势分析¶

    3.1 降低耦合度¶

    新设计通过引入属性管理器，将物理属性的存储与网格单元解耦。网格单元不再直接存储物理属性，而是通过属性管理器按需获取。

    3.2 明确职责划分¶

    • PropertyManager 负责物理属性的存储和访问
    • SinglePhaseProperties 专注于物理计算
    • FVMDiscretizer 负责数值离散化

    3.3 提高可测试性¶

    每个类的职责明确，可以独立进行单元测试： - 可以单独测试 PropertyManager 的属性初始化和访问功能 - 可以单独测试 SinglePhaseProperties 的物理计算功能 - 可以单独测试 FVMDiscretizer 的离散化功能

    3.4 增强可扩展性¶

    • 易于添加新的物理属性类型
    • 易于支持复杂的非均匀属性场
    • 易于扩展到其他类型的网格（如非结构化网格）

    3.5 内存优化¶

    对于均匀属性，可以直接存储单个值而不是整个数组，节省内存。

    4. 实施计划¶

    4.1 第一阶段：重构属性管理¶

    1. 创建 PropertyManager 类
    2. 将属性存储和访问逻辑移到 PropertyManager
    3. 修改 SinglePhaseProperties 类，移除直接操作网格单元的代码

    4.2 第二阶段：更新离散化器¶

    1. 修改 FVMDiscretizer 类，使用新的属性访问方式
    2. 移除对网格单元属性的直接访问

    4.3 第三阶段：测试和验证¶

    1. 编写单元测试验证新设计的正确性
    2. 运行现有示例确保功能正常
    3. 性能测试确保没有性能下降

    5. 风险评估和缓解措施¶

    5.1 性能风险¶

    按需获取属性可能比直接访问单元属性稍慢。缓解措施： - 使用缓存机制优化频繁访问的属性 - 对热点代码进行性能优化

    5.2 兼容性风险¶

    修改类接口可能影响现有代码。缓解措施： - 提供适配器模式保持向后兼容 - 逐步迁移，不破坏现有功能

    5.3 复杂性风险¶

    引入新的抽象层可能增加代码复杂性。缓解措施： - 提供清晰的文档和示例 - 保持接口简洁易用
